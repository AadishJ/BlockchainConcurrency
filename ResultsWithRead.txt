Q. Why can we use Promise.all for concurrency?

Ans. So the tasks we are performing are I/O bound task and do not require that much cpu so when we use Promise.all() it creates a event loop which is not really a loop but it just executes all the request without waiting for them to finish after all the request have been created then the waiting starts for all of them to finish and when all promises succeed it resolves the promise and we can move on with the code

Q. Why will using multithreading not be that benificial?

Ans. The I/O operation we are performing are non-blocking operations which means they do not directly require any cpu so they can be executed concurrently even with a single thread. The bottleneck is network latency (waiting for the blockchain), not CPU processing.

Q. Whats an event loop?

Ans. The event loop is the mechanism in JavaScript that handles asynchronous operations (like reading a file, making a network request, or waiting for user input) without blocking the execution of our program. Its single threaded. 

Q. Does read request require mining?

Ans. No read requests do not require any mining as it is just returning the state of blockchain so it does not create any transaction.
